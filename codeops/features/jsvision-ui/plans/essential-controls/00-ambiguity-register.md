## Plan-level decisions (PA-NN)

| PA # | Category | Question | Options Considered | Decision | Status |
|------|----------|----------|--------------------|----------|--------|
| PA-1 | Architecture · command emission | How does a leaf control (e.g. `Button`) emit a typed command, given chrome gets a `createApplication`-injected seam but app-authored controls in a `Window` have no attach point? | (a) **expose `emit(command, arg?)` on the dispatch envelope** (`DispatchEvent`) — surface the existing `RouteContext.emitCommand` (`dispatch.ts:31`) so a focused control's `onEvent` emits directly (TV-faithful: button→command to owner); `Button` supports both `command` + `onClick`; small additive intra-ui view/event-contract change (like RD-05 `setCapture`/`onFrame`); (b) `onClick` callback only, command emission is the app's job (`onClick: () => loop.emitCommand(...)`) — rejected: leaves `Button.command` unwired, AC-3 satisfied only via the app; (c) a per-control attach seam like `Window.manager` — rejected: heavy, every control would need attaching | **(a) additive `ev.emit()` on `DispatchEvent`** | ✅ Resolved (user) |
| PA-2 | Behavior · invalid-on-focus-leave | What happens when a blocking validator (`range`/`lookup`) fails as focus leaves the `Input` — TV traps focus via the Dialog's `valid()` sweep, but RD-06 has no `Dialog` (RD-11)? | (a) **expose `valid(): boolean` + a themable `invalid` state, no focus-trap** — on focus-leave run the blocking validator, set `invalid`; Tab/click-away proceeds; the modal focus-trap is deferred to RD-11's `Dialog` (registered in `DEFERRED.md`); avoids a focus-veto primitive in RD-06; (b) focus-trap now (veto Tab/click-away until valid) — rejected: needs a new `canBlur`/focus-veto primitive in RD-04, and a trapped bare field with no modal context confuses | **(a) `valid()` + `invalid` state, no trap; trap → RD-11** | ✅ Resolved (user) |
| PA-3 | Data · cluster value shape | How do `CheckGroup`/`RadioGroup` expose their bound value (the two-way signal, AR-100)? | (a) **`CheckGroup` ⇒ `Signal<boolean[]>`** (one flag per item), **`RadioGroup` ⇒ `Signal<number>`** (selectedIndex) — explicit, reactive-friendly, per-item read/write; (b) bitmask `Signal<number>` for check (TV `value ^= 1<<i`) — rejected: bit-twiddly, un-idiomatic in a reactive API | **(a) `boolean[]` / `number`** | ✅ Resolved (user) |
| PA-4 | File layout | Where do controls live + how re-exported? | (a) **`packages/ui/src/controls/` subsystem** — `text.ts`/`label.ts`/`button.ts`/`input.ts`/`cluster.ts`/`check-group.ts`/`radio-group.ts`, validators under `controls/validators/` (`types.ts`/`filter.ts`/`range.ts`/`lookup.ts`), one barrel `index.ts`, **explicit named re-exports** from `src/index.ts`; each file ≤ 500 lines; (b) scatter across existing dirs — rejected (breaks the per-subsystem pattern) | **(a) `src/controls/` subsystem** | ✅ Resolved (AR-102; dominant) |
| PA-5 | Theme · role byte sourcing | How are the exact `cpGrayDialog` control-role colors pinned, given the two-level `cpGrayDialog`→`cpAppColor` indirection is error-prone (a recon decode disagreed with the hand-verified value)? | (a) **the theme-role spec test computes each role from `app.h:142` `cpAppColor[ cpGrayDialog[slot] ]` read directly from source** (the immutable oracle derives from the source, per the fidelity directive); buttons **reuse** the existing `button` (`0x20` black/green) + `buttonFocused` (`0x2F` white/green) roles (verified to match TV button normal/selected), only **adding** the new roles; (b) freeze an agent's hand-decode — rejected: unreliable (proven), violates source-as-oracle | **(a) pin role bytes from `app.h` in the spec oracle; reuse `button`/`buttonFocused`** | ✅ Resolved (AR-97; source-determined) |
| PA-6 | Cluster · column layout | Single-column or TV's multi-column flow (`size.y` rows per column)? | (a) **single-column in v1** — one item per row, `size.y` = item count (the RD says "a vertical column"); TV's multi-column flow is **deferred** (`DEFERRED.md`); the per-item cell geometry (5-cell box + label) is still faithful; (b) full multi-column now — rejected: the RD scoped a single column, multi-column adds column-math + ←→ nav not needed for v1 | **(a) single-column; multi-column deferred** | ✅ Resolved (AR-96; dominant) |
| PA-7 | Button · default + activation | How does a `default` button respond to Enter, and how is a focused button activated? | (a) **`Space`/`Alt-hotkey` activate the focused button; a `default:true` button is `postProcess` and activates on `Enter` if unconsumed** (TV `cmDefault` broadcast + `bfDefault`; mirrors the RD-05 `postProcess` accelerator pattern, `statusline.ts`); (b) app-level default handler — rejected: re-implements what `postProcess` already gives | **(a) Space/hotkey focused; Enter→`postProcess` default** | ✅ Resolved (source-determined) |
| PA-8 | Button · shadow rendering | Block-glyph shadow or `ctx.shadow` darkening? | (a) **faithful TButton block-glyph shadow** — right column + bottom row drawn with `▄`(0xDC)/`█`(0xDB)/`▀`(0xDF) in a shadow style (`tbutton.cpp` `drawState`), the clickable area excluding the shadow row/col; (b) `ctx.shadow` cell-darkening — rejected: that's the window/desktop drop-shadow, not TV's button shadow glyphs | **(a) TButton block-glyph shadow** | ✅ Resolved (source-determined) |
| PA-9 | Glyph · radio marker width | What filled-marker glyph for the selected radio (`TRadioButtons` uses CP437 `0x07`, East-Asian-ambiguous)? | (a) **prefer an unambiguous-narrow code point** for the filled marker (follows the established close-glyph `[×]` precedent / the CLAUDE.md EA-ambiguous note); brackets `( )` + checkbox `X` are ASCII-narrow; the exact marint pinned in the spec oracle at implementation; (b) raw CP437 `0x07`→`◉` (U+25C9, ambiguous) — rejected: misaligns like the `■` close glyph did | **(a) unambiguous-narrow filled marker** | ✅ Resolved (source + CLAUDE.md; dominant) |
| PA-10 | Label · focus-link | How does `Label` link to + reflect a control? | (a) **`Label` holds a direct reference to its target `View` (`link`)**; a click (via hit-test) or its `~hotkey~` (caught post-process, since the Label is not focusable) focuses the link via the **additive `ev.focusView` envelope accessor** (bundled with PA-1's `ev.emit`, sourced from the `RouteContext` focus manager); the label paints `labelSelected` while the link is `state.focused` (observed reactively via `bind`) — faithful to `tlabel.cpp:91-98` `focusLink`; (b) a string-id lookup — rejected: indirection with no registry in scope | **(a) direct `View` link + `ev.focusView` accessor** | ✅ Resolved (source-determined) |
| PA-11 | Input · lean scope detail | What exactly does the lean `Input` include? | (a) **cursor + `firstPos` horizontal scroll + `◄`/`►` edge arrows + `maxLength` + two-way `value` signal + validator hook + the printable/Backspace/Delete/Home/End/←/→ key set** (`tinputli.cpp`); **selection + cut/copy/paste clipboard deferred** (AR-94, `DEFERRED.md`); (b) add selection now — rejected (AR-94) | **(a) lean edit set; selection/clipboard deferred** | ✅ Resolved (AR-94; source-determined) |
| PA-12 | Validator · type shape | What is the `Validator` shape + how do `filter`/`range`/`lookup` produce one? | (a) **`Validator = { isValidInput(s: string): boolean; isValid(s: string): boolean; error?: string }`** (mirrors TV `isValidInput`/`isValid`); `filter(chars)`/`range(min,max)`/`lookup(list)` are factory functions returning a `Validator`; (b) a class hierarchy — rejected: the component map reframes the `TValidator` hierarchy as small typed units | **(a) `Validator` object + factory functions** | ✅ Resolved (AR-95; source-determined) |
| PA-13 | Demo · host | How are the controls hosted in `demo:controls`, with no `Dialog` (RD-11) yet? | (a) **a plain `Group`/`Window`** holding `Label`+`Input`+`CheckGroup`+`RadioGroup`+`Button`; each control self-colors via its roles (no separate gray container — the gray `Dialog` is RD-11); a headless dispatch walkthrough (Tab focus, validator rejection, toggle/select, Button emit) with an ASCII frame per step + an e2e asserting narration + key glyphs; (b) build a stand-in gray panel — rejected: premature, RD-11 owns the Dialog | **(a) plain Window/Group host; controls self-color** | ✅ Resolved (AR-98; dominant) |
| PA-14 | Text · wrap + align | `TStaticText` wrap + alignment scope? | (a) **word-wrap on spaces, left-aligned** (faithful `tstatict.cpp` word-wrap); TV's leading-`0x03` center marker is **out of v1** (optional later) — RD-06 `Text` is left-aligned wrapped content; (b) full center/right alignment now — rejected: not required, adds the `0x03` convention | **(a) word-wrap, left-aligned; center out** | ✅ Resolved (dominant) |
